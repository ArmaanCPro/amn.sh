## Abstract

Automatic memory management is one of the defining features of modern programming languages. By delegating memory reclamation to the runtime system, languages such as Java, Go, and C# eliminate entire classes of memory safety errors common in manual allocation environments like C and C++. However, this convenience comes with significant design tradeoffs. Garbage collectors influence runtime performance, memory layout, compiler implementation, and system latency characteristics.

This paper surveys several fundamental garbage collection strategies—reference counting, mark–sweep, mark–compact, copying collectors, generational garbage collection, and concurrent collection techniques. We analyze their algorithmic behavior and examine the practical tradeoffs faced by language and runtime designers. In particular, we focus on how these collectors interact with compilers, including root identification, write barriers, heap layout, and pause-time considerations.

---

## 1. Introduction

Memory management is a central problem in programming language implementation. In systems programming environments such as C and C++, memory allocation is explicitly controlled through mechanisms such as `malloc`/`free` or `new`/`delete`. While this model offers maximal control and predictable performance, it also introduces the possibility of severe bugs including use-after-free, double-free, and memory leaks.

Garbage collection (GC) offers an alternative model: memory is automatically reclaimed when objects are no longer reachable from the program's roots. While conceptually simple, implementing efficient garbage collectors is a deeply complex systems problem. The collector must:

- Identify **live objects**
- Reclaim **unreachable memory**
- Minimize **runtime overhead**
- Avoid **excessive pause times**

Different GC algorithms approach these challenges in different ways, each introducing tradeoffs in latency, throughput, memory overhead, and implementation complexity.

This paper examines the most widely used GC strategies and discusses how they affect both **runtime systems** and **compiler design**.

## 2. Reference Counting

Reference counting is one of the simplest garbage collection techniques. Each object maintains a counter representing the number of references pointing to it. When the reference count drops to zero, the object can be immediately reclaimed.

### Algorithm

Whenever a pointer is assigned, the runtime increments the target object's reference count and decrements the previous object's count.

```pseudo
function assign(ptr, new_object):
    old_object = ptr.value

    if new_object != null:
        new_object.refcount += 1

    ptr.value = new_object

    if old_object != null:
        old_object.refcount -= 1
        if old_object.refcount == 0:
            free(old_object)
```

### Advantages

Reference counting offers several desirable properties:

* **Immediate reclamation** of unused objects
* **Low pause times** since no global heap scan is required
* Straightforward conceptual model

These characteristics make reference counting attractive in interactive environments.

### Limitations

However, reference counting suffers from a fundamental weakness: **cyclic references**.

Consider two objects referencing each other but otherwise unreachable from program roots. Their reference counts never reach zero, causing a memory leak.

Additionally, reference counting introduces **constant runtime overhead** on pointer assignments, which can significantly affect performance in pointer-heavy programs.

### Real-World Usage

Reference counting is widely used in systems such as:

* Python's primary GC mechanism
* Swift's ARC (Automatic Reference Counting)

However, these systems typically supplement reference counting with **cycle detection algorithms**.

## 3. Mark–Sweep Collection

Mark–sweep collection is one of the earliest tracing garbage collection algorithms. Instead of tracking reference counts, the collector periodically traverses the object graph to identify reachable objects.

The algorithm proceeds in two phases:

1. **Mark phase** — traverse reachable objects
2. **Sweep phase** — reclaim unmarked memory

### Algorithm

```pseudo
function mark(root):
    if root.marked:
        return

    root.marked = true

    for child in root.references:
        mark(child)

function mark_sweep(heap, roots):
    for root in roots:
        mark(root)

    for object in heap:
        if object.marked:
            object.marked = false
        else:
            free(object)
```

### Advantages

Mark–sweep collectors eliminate the cyclic reference problem present in reference counting. They also avoid per-assignment overhead.

### Limitations

The primary drawback is **heap fragmentation**. Since objects remain in place after collection, the heap becomes increasingly fragmented over time.

Another drawback is **stop-the-world pauses**, where program execution must halt while the collector scans memory.

## 4. Mark–Compact Collection

Mark–compact collectors address the fragmentation problem of mark–sweep collectors by relocating live objects after marking.

After identifying reachable objects, the collector **moves them into contiguous memory**, updating references accordingly.

### Algorithm Outline

```pseudo
mark_live_objects()

new_address = heap_start

for object in heap:
    if object.marked:
        object.forwarding = new_address
        new_address += object.size

for object in heap:
    if object.marked:
        move(object, object.forwarding)
```

### Advantages

Mark–compact collectors provide:

* **Reduced fragmentation**
* Improved **memory locality**

Better locality can significantly improve cache performance.

### Limitations

However, object relocation introduces additional complexity:

* Pointer updates must occur throughout the heap
* Stack references must be rewritten
* Compiler cooperation is required

Because of these requirements, mark–compact collectors typically rely on **precise root information provided by the compiler**.

## 5. Copying Collectors

Copying collectors divide the heap into two equal regions:

* **From-space**
* **To-space**

Allocation occurs in from-space. During garbage collection, live objects are copied into to-space, compacting them automatically. Once copying completes, the spaces are swapped.

This approach was popularized by Cheney's algorithm.

### Cheney's Algorithm

```pseudo
function cheney_gc():
    swap(from_space, to_space)

    scan = to_space.start
    free = to_space.start

    for root in roots:
        root = copy(root)

    while scan < free:
        for ref in scan.references:
            ref = copy(ref)
        scan += scan.size
```

### Advantages

Copying collectors offer several strong properties:

* **Automatic compaction**
* **Fast allocation** (simple bump pointer)
* **Improved cache locality**

These characteristics make copying collectors particularly attractive for short-lived objects.

### Limitations

The most significant drawback is **space overhead**, since half the heap must remain unused during copying.

This cost can be substantial in memory-constrained environments.

## 6. Generational Garbage Collection

Empirical observations about program behavior led to the **generational hypothesis**:

> Most objects die young.

Generational collectors exploit this by dividing objects into multiple generations:

* **Young generation**
* **Old generation**

Young objects are collected frequently using fast copying collectors, while older objects are collected less often.

### Generational Layout

```
+-------------------+
| Young Generation  |
|  Eden             |
|  Survivor Spaces  |
+-------------------+
| Old Generation    |
+-------------------+
```

### Benefits

Generational collectors significantly reduce GC overhead because:

* Most garbage is collected quickly
* Old objects are rarely scanned

This approach dramatically improves overall throughput.

### Write Barriers

However, generational collectors introduce an important complexity: **tracking references from old objects to young objects**.

This requires **write barriers**, small pieces of code inserted by the compiler whenever pointers are modified.

Example write barrier:

```pseudo
function write_barrier(obj, field, value):
    obj.field = value
    if obj in old_generation and value in young_generation:
        remember_set.add(obj)
```

The compiler or JIT must cooperate with the runtime to ensure these barriers execute correctly.

## 7. Concurrent and Incremental Garbage Collection

Traditional collectors often require **stop-the-world pauses**, where application execution halts while the collector runs. For interactive systems and large heaps, these pauses can become unacceptable.

Concurrent and incremental collectors attempt to reduce pause times by performing GC work alongside program execution.

### Incremental Collection

Incremental collectors divide collection work into small steps interleaved with program execution.

This reduces pause times but increases overall runtime complexity.

### Concurrent Collection

Concurrent collectors run simultaneously with the application on separate threads.

However, concurrency introduces new challenges:

* The program may modify objects while they are being scanned
* The collector must maintain heap consistency

To address this, collectors use techniques such as:

* **Tri-color marking**
* **Snapshot-at-the-beginning algorithms**
* **Read/write barriers**

These techniques significantly increase implementation complexity.

## 8. Implications for Compiler and Runtime Design

Garbage collection is not purely a runtime problem. Efficient collectors require significant **compiler cooperation**.

### Root Identification

The collector must identify **roots**, including:

* Stack variables
* CPU registers
* Global variables

Precise collectors require the compiler to generate **stack maps** describing pointer locations at GC-safe points.

### Write Barriers

Generational and concurrent collectors require the compiler to emit **write barriers** on pointer updates. These barriers must be carefully optimized to avoid excessive overhead.

### Safe Points

Programs must periodically reach **GC safe points**, where execution can pause and the collector can safely inspect program state.

The compiler typically inserts safe points at:

* Loop back edges
* Function calls
* Allocation sites

### Escape Analysis

Modern compilers may also perform **escape analysis** to determine whether objects can remain on the stack rather than the heap. This optimization can significantly reduce GC pressure.

#### Insights for Compiler and Runtime Designers

1. **Allocation patterns matter** — Short-lived objects benefit most from generational or copying collectors. Languages or engines that create many transient objects can optimize for minor collections to improve throughput.

2. **Pause time vs throughput tradeoff** — Interactive applications (games, GUIs) favor low-latency collectors like concurrent or generational GC. Batch-processing workloads may tolerate higher pause times from mark–compact or mark–sweep collectors.

3. **Heap layout and memory locality** — Copying and generational collectors improve cache performance by compacting live objects. Compilers can assist by organizing hot vs cold data in memory.

4. **Compiler cooperation is crucial** — Write barriers, safe points, and precise root maps reduce GC overhead. Early integration of compiler/runtime design decisions is recommended.

5. **Experimentation is valuable** — Implementing small-scale GC benchmarks, even with toy workloads, helps identify bottlenecks in allocation and collection strategies.

## 10. Conclusion

Garbage collection remains one of the most challenging components of modern language runtimes. Different GC strategies present tradeoffs across latency, throughput, memory usage, and implementation complexity.

Simple approaches such as reference counting provide predictable reclamation but struggle with cyclic structures. Tracing collectors such as mark–sweep eliminate these issues but introduce pause times and fragmentation. More advanced collectors—generational, concurrent, and incremental—attempt to balance these tradeoffs but require increasingly sophisticated runtime and compiler cooperation.

Ultimately, no single garbage collection algorithm is optimal for all workloads. Runtime designers must carefully select or combine strategies depending on the language's goals, target workloads, and performance requirements.

As managed languages continue to dominate large-scale software development, improvements in garbage collection algorithms and compiler integration remain an active and important area of systems research.
